# 百日实践营学员管理系统 · 产品需求文档(PRD)

> 版本:v1.0
> 文档用途:本文档面向 AI 编程工具或开发者,可据此独立完成系统设计与开发。
> 项目背景:服务于「AI时代为人父母·百日实践营」(12 周 / 100 天周期)的日常运营管理。

---

## 1. 产品概述

### 1.1 一句话定义
一个真正部署上线、数据存云端的 Web 系统,供**组长**代学员录入日常学习情况、供**讲师**查看与点评、供**管理员**统计全局数据,把百日营 12 周运营动作沉淀为可追踪、可管理的线上工具。

### 1.2 要解决的核心痛点
当前运营靠微信群 + 打卡小程序 + 人工统计,存在三个问题:
1. **数据散乱**:打卡、作业、出勤分散在不同工具,无法统一查看每位学员的全程轨迹。
2. **点评无沉淀**:讲师点评散落在群聊,组长和后续无法回溯。
3. **统计靠手工**:出勤率、打卡率、作业提交频率全靠人工汇总,费时易错。

### 1.3 产品目标
| 目标 | 可衡量指标 |
|------|-----------|
| 统一数据入口 | 组长在一个系统内完成全部学员日常录入 |
| 点评可追溯 | 100% 讲师点评有记录、组长可查看 |
| 统计自动化 | 出勤率/打卡率/作业提交率自动计算,可导出 |
| 咨询有序化 | 一对一咨询线上预约,规则自动约束,无重复约 |
| 见证可沉淀 | 学员「看见时刻」结构化留存,支持精选发布 |

### 1.4 设计原则
- **轻录入**:组长每日录入动作尽量少、批量化,降低使用负担。
- **规则前置**:关键约束(如咨询限 1 次)由系统强制,不靠人工把关。
- **权限清晰**:每个角色只看到自己该看的内容。
- **不堆砌**:只做运营真实需要的功能,不加冗余模块。

## 2. 角色与权限

### 2.1 角色定义

| 角色 | 数量 | 是否登录 | 核心职责 |
|------|------|---------|---------|
| 学员 | 44–62 人 | **否** | 不直接使用系统,其数据由组长代录 |
| 组长 | 7 位(每组 1 位) | 是 | 代本组学员录入日常情况、代约咨询、查看本组看板 |
| 讲师 | 4 位 | 是 | 查看所有学员内容、点评、发布可约时段、管理预约 |
| 管理员 | 1 位 | 是 | 全局统计、见证墙管理、基础数据维护 |

### 2.2 权限矩阵

| 功能 | 组长 | 讲师 | 管理员 |
|------|:----:|:----:|:------:|
| 登记出席打卡 | 本组 | — | 全局(可代操作) |
| 录入作业/收获/日记/行动分享 | 本组 | — | 全局 |
| 查看学员内容 | 本组 | **全部** | 全局 |
| 点评学员内容 | — | **全部** | — |
| 查看点评 | 本组 | 全部 | 全局 |
| 发布可约咨询时段 | — | 本人 | — |
| 预约咨询 | 本组学员 | — | — |
| 管理预约(确认/完成) | — | 本人被约的 | 全局查看 |
| 投稿见证墙 | 本组 | — | — |
| 见证墙精选/发布 | — | — | **是** |
| 出勤率/打卡率统计 | 本组 | — | **全局** |
| 账号/分组/课程维护 | — | — | **是** |

> **数据权限铁律**:组长只能读写**自己组**的学员数据;讲师可**跨组只读**全部学员数据并点评;管理员看全局。系统须在接口层强制校验,不能仅靠前端隐藏。

## 3. 功能需求(按角色)

### 3.1 组长端

#### F-L1 每日出席打卡登记
- 进入本组「今日打卡」页,以学员列表形式逐人标记:**已到 / 未到**。
- 支持**补登**:可切换到任意历史日期补录(对应方案「允许补卡」)。
- 打卡可附简短备注(可选)。
- 一键查看本组今日打卡完成进度(如 6/7)。

#### F-L2 作业提交登记
- 按「课程/周次 + 学员」录入作业状态:**已交 / 未交**。
- 已交可附作业内容文本或备注(对应「作业提交频率登记」)。
- 列表清晰展示本组每位学员的作业提交情况。

#### F-L3 单次课程个人收获录入
- 选择某一次课程,为指定学员录入其「本节课的个人收获」(文本)。
- 一次课程一位学员对应一条收获记录。

#### F-L4 内省日记录入
- 为学员录入「内省日记」(对应方案的每日觉察动作 / 内省)。
- 按日期 + 学员归档。

#### F-L5 生活行动建议分享录入
- 为学员录入「生活行动建议分享」(对应生活行动线)。
- 按日期 + 学员归档。

#### F-L6 代学员预约咨询
- 浏览各讲师**已发布的可约时段**(显示讲师、日期、时间段、是否已被占用)。
- 选择一位本组学员 + 一个空闲时段提交预约。
- **系统强制**:该学员若已有过预约记录,则无法再次提交(见 4.1)。
- 可查看 / 取消本组学员尚未开始的预约。

#### F-L7 见证墙投稿
- 选择本组学员,提交一条「看见时刻」:一句话看见 + 可选一张图片。
- 设置署名方式:**匿名 / 署名**。
- 投稿进入待审核状态,由管理员精选发布(见 3.3)。

#### F-L8 本组汇总看板
- 一屏展示本组关键数据:本周打卡率、各学员作业提交情况、出勤汇总、断卡提醒(连续 N 天未打卡高亮)。
- 查看本组学员收到的讲师点评。

---

### 3.2 讲师端

#### F-T1 学员内容浏览
- 查看**全部**学员提交的:作业、单次课程收获、内省日记、生活行动分享。
- 支持筛选:按组别、按学员、按课程/周次、按日期、按「是否已点评」。
- 默认优先展示「未点评」内容,方便讲师认领(对应「谁有空谁点评」)。

#### F-T2 内容点评
- 对任意一条学员内容撰写点评(文本)。
- 点评一旦提交,**对应组长可见**;同一条内容可被点评(规则见 4.2)。
- 显示点评人(哪位讲师)与点评时间。

#### F-T3 发布可约咨询时段
- 讲师自行新增/删除自己的可约时段:选择日期 + 起始时间,系统按 **30 分钟**为单位生成时段。
- 已被预约的时段不可删除,显示为「已约」。

#### F-T4 咨询预约管理
- 查看自己被预约的全部咨询:学员姓名、所属组、时间。
- 操作:**确认预约 / 标记已完成**。

---

### 3.3 管理员端

#### F-A1 出勤率统计
- 按**单次课程**统计整体出勤率(出席人数 / 应到人数)。
- 可查看每节课明细,支持**导出**(CSV/Excel)。

#### F-A2 全营数据总览
- 总览:全营打卡率、作业提交频率、各组对比。
- 时间维度可切换(本周 / 本月 / 全程)。

#### F-A3 见证墙管理
- 查看组长提交的全部「看见」投稿。
- 精选 / 发布 / 撤下;尊重投稿时的匿名/署名设置。
- 已发布内容形成「见证墙」展示页。

#### F-A4 基础数据维护
- 管理:学员名单与分组、组长账号、讲师账号、课程/周次表。
- 重置密码、启用/停用账号。

## 4. 关键业务规则

### 4.1 一对一咨询规则(最高优先级)
- 流程:**讲师发布可约时段 → 组长为学员挑选时段 → 提交预约 → 讲师确认 → 完成后标记**。
- **每位学员全程仅限预约 1 次**。系统在提交时校验该学员的历史预约记录(含已完成的),已有则拒绝并提示。
- 每次咨询固定 **30 分钟**。
- 一个时段只能被一位学员占用,占用后对其他组长不可选(并发下需服务端加锁,防止两人同时抢到同一时段)。
- 取消预约后,该时段释放;但学员的「已用 1 次」额度是否恢复需明确——**默认:取消未确认的预约可恢复额度,已确认/已完成的不恢复**(开发时按此实现,后续可由管理员调整)。

### 4.2 点评规则
- 所有讲师可见所有学员内容,任意讲师均可点评(不固定分配讲师与组)。
- 一条内容允许多位讲师分别点评(便于「谁有空谁点」),每条点评独立记录点评人。
- 点评对**该内容所属学员的组长**可见,对其他组长不可见。

### 4.3 出勤与打卡规则
- 出勤率按**单次课程**计算,独立于每日打卡。
- 每日打卡允许补登,不因断卡影响数据完整性。
- 「断卡提醒」:连续 3 天未打卡的学员在组长看板高亮(阈值可配置)。

### 4.4 见证墙规则
- 投稿默认进入待审核;仅管理员精选后才公开。
- 严格遵循匿名/署名设置:匿名投稿在公开页不得显示学员姓名。

### 4.5 数据权限规则
- 见 2.2「数据权限铁律」,须在服务端接口层强制校验。

## 7. 核心用户流程

### 7.1 组长每日录入流程
```
登录 → 本组工作台 → 今日打卡(逐人标到/未到)
     → 作业登记(本次课谁交了)
     → 录入学员收获/内省/行动分享
     → 查看看板,对断卡学员线下提醒
```

### 7.2 讲师点评流程
```
登录 → 工作台默认显示「未点评」内容流
     → 选一条内容 → 撰写点评 → 提交
     → 该内容标记已点评,对应组长可见
```

### 7.3 咨询预约流程(强约束)
```
讲师:发布可约时段(按 30 分钟生成)
  ↓
组长:浏览各讲师空闲时段 → 选学员 + 选时段 → 提交
  ↓
系统校验:该学员是否已有有效预约?
  ├─ 是 → 拒绝,提示「该学员已预约过,每人限 1 次」
  └─ 否 → 锁定时段(并发加锁)→ 创建预约(pending)
  ↓
讲师:确认(confirmed)→ 咨询后标记完成(completed)
```

### 7.4 见证墙流程
```
组长:选学员 → 填一句话看见 + 配图 + 选匿名/署名 → 提交(pending)
  ↓
管理员:审核 → 精选发布(published,遵循匿名设置)
  ↓
见证墙公开展示页呈现
```

## 8. 技术栈建议

> 以下为推荐方案,目标是**易部署、低维护成本、适合 60 人规模**。开发者可按自身熟悉度替换。

| 层 | 建议 | 理由 |
|----|------|------|
| 前端 | React + Vite + Tailwind CSS,或 Next.js | 组件化、移动端友好(组长多用手机录入) |
| 后端 | Node.js(NestJS/Express)或 Python(FastAPI) | 生态成熟、开发快 |
| 数据库 | PostgreSQL | 支持唯一约束/事务,满足咨询并发加锁需求 |
| 认证 | JWT + 角色鉴权中间件 | 简单可靠,支持按 role/group 校验 |
| 文件存储 | 对象存储(见证墙图片) | 与数据库解耦 |
| 部署 | 一体化平台(如 Vercel/Render + 托管数据库,或国内云主机 + Docker) | 上线快、运维轻 |

> **移动端优先**:组长主要在手机上录入,前端须做响应式,关键录入页要适配小屏、操作步骤少。

## 9. 非功能需求

### 9.1 安全
- 密码加盐哈希存储,绝不明文。
- 所有接口做角色 + 数据范围(组)双重校验,前端隐藏不可作为唯一防线。
- 见证墙图片及学员信息涉及隐私,匿名设置必须严格生效。
- 内容使用授权:学员报名时已签授权书,系统对匿名投稿不得反查暴露姓名。

### 9.2 性能与规模
- 目标并发:几十人级别,无高并发压力;但**咨询抢时段须保证一致性**(事务 + 唯一约束)。
- 常规列表查询响应 < 1 秒。

### 9.3 可用性
- 移动端友好,组长录入路径尽量 3 步内完成。
- 关键操作有成功/失败提示;误操作(如重复约)有清晰文案。

### 9.4 数据备份
- 数据库每日自动备份,保留至少 30 天。

### 9.5 可维护性
- 管理员可自助维护学员/账号/课程,无需开发介入。
